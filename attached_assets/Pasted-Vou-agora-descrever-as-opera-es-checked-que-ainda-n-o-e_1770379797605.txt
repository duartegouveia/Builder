Vou agora descrever as operações checked, que ainda não estão definidas, ou seja as operações que se vão efetuar sobre o conjunto de registos que foram assinalados na lista.

Do ponto de vista lógico as operações "Multiview, Multiedit, Multidelete e Multicopy" são muito parecidas estruturalmente. As operações "Group Edit" e "Merge" são estruturalmente diferentes.

Todas estas operações partilham da lógica já anteriormente descrita para as operações de linha, no sentido em que têm um painel no fundo com botões, sendo que o da direita é o fechar, e alinhado à esquerda estão botões específicos da operação.
Tal como as operações de linha, estas operações podem ser geradas para "dialog", "right" ou "bottom". 

O que as operações "Multiview, Multiedit, Multidelete" e "Multicopy" têm em comum, é a possibilidade de terem multiplos paineis de visualização: 1, 2, 3 ou 4. Cada um deles com a sua estrutura de navegação (primeiro, anterior, #número de #total, próximo, último), que deve ser idêntica em aspeto à navegação entre páginas na vista table e cards, mas neste caso aplicado aos registos checked, em vez de ser às páginas.
Na barra do fundo esquerda deve existir uma select a permitir escolher "1 panel", "2 panels", "3 panels", "4 panels". Por omissão a escolha é 2 panels. As operações multi não fazem sentido ser chamadas se não existirem pelo menos 2 registos checked.
Também na inicialização, os paineis devem mostrar registos distintos, entre o select, seguindo a ordem que foram checked, ou seja no painel 1 mostra o primeiro checked, no painel 2 o segundo, etc. O utilizador pode escolher livremente que registo checked mostra no primeiro painel, no segundo, etc. Esta regra tem uma exceção para o multiedit. No caso do multiedit, por serem campos editáveis, faz confusão ter dois campos que podem ser editados para um mesmo registo, pelo que, não deixa ver uma página que já foi escolhida para edição num painel anterior. Na navegação, se detectar que já existe um painel em edição para aquele registo, ou salta para o registo seguinte (no caso anterior ou próximo), ou se não for possível navegar nessa direção, não deixa mudar de registo. Talvez mostrar um toast a explicar porque não esta a mover-se.
Implementa a lógica dos toasts de forma genérica para poder ser utilizada noutras zonas do código de forma independente.

Há mais um detalhe importante, que é comum às operações "Multiview, Multiedit, Multidelete" e "Multicopy". Apesar de cada um dos paineis aparentar ser independente um do outro, há um detalhe de dependência entre eles. Os vários campos de cada registo podem ter tamanhos distintos (em altura). Nas operações multi, quando temos vários paineis (>=2), todos os campos passam a ocupar a mesma altura, que campo a campo deve ser a altura do que ocupa mais espaço. Isto tem de ser feito dinâmicamente porque o tamanho do campo depende do seu conteúdo específico que naturalmente muda de registo para registo. Sempre que num painel se navega para outro registo o cálculo das alturas para totos os campos de todos os paineis tem se recalculado, tendo por base o tamanho que cada um deles necessita, ou seja, primeiro é necessário deixar que os campos reponham o seu tamanho (height) natural, para depois os reajustar pela altura máxima dos campos em concreto escolhidos nos paineis visíveis. 

A operação Multiview apenas permite mostrar registos, pelo que implementa o caso genérico acima explicado. O conteúdo de cada painel é o que é usado para a operação "View", pelo que na medida do possível, deve ser usadaa mesma função para tratar os diversos kinds de dados. Ao contrário do que acontece na operação view em que existem botões para passar para Edit, Copy, Delete, aqui existe apenas o escolher o número de paineis, o navegar entre registos e o fechar.

A operação Multiedit tem um botão específico para "Gravar" comum aos vários paineis. O conteúdo de cada painel é o que é usado para a operação "Edit", pelo que na medida do possível, deve ser usadaa mesma função para tratar os diversos kinds de dados. Ou seja, ao gravar, guarda as mudanças efetuadas em todos. Após gravar, fecha a operação Multiedit.

A operação Multidelete usa como base para os dados no painel a operação "View", tal como acontece com o "Multiview". A operação Multidelete tem um botão específico para "Apagar Todos". Após "Apagar Todos", fecha a operação Multidelete.

Na operação Multicopy usa como base para os dados no painel a operação "View", tal como acontece com o "Multiview". A operação Multicopy, tal como existe na operação Copy, existe um input para poder indicar o número inteiro de cópias (>=1) e o botão "Gerar Cópias", e ao carregar em "Gerar Cópias" faz o mesmo que faz a operação "Copy" para um registo, mas desta vez para vários registos.

As operações "Group Edit" e "Merge" têm uma interface diferente.

O propósito da operação "Group Edit" é ter uma forma fácil de definir valores específicos para um ou mais campos, para todos os registos checked. 
A operação Group Edit não tem os múltiplos paineis, mas apenas um. Para cada campo precede a interface que seria esperável para uma operação edit desse kind, com uma checkbox. A parte editável daquele kind só fica visível quando a checkbox é assinalada. Mas o nome do campo fica sempre visível. Nessa altura o utilizador pode definir o valor pretendido para aquele kind.
O utilizador pode escolher mais do que um campo para fazer a operação "Group Edit".
No fundo, alinhado à esquerda está o botão "Gravar para Todos", que ao ser carregado, vai alterar apenas os campos assinalados, para apenas os registos checked que estiveram na origem da chamada a esta operação. Campos que não tenham uma versão editavel, como o caso dos kind "id" e "relation", aparecem na lista, mas não têm checkbox para não dar ao utilizador a esperança de poderem ser editados por esta operação. 
Quando a checkbox para um campo não está assinalada, na zona que seria para o campo editável, aparece uma síntese em formato de texto, com os valores que os registos checked têm atualmente para aquele campo, como histograma, ou seja, sem repetidos, e com a contagem (#n) à frente de cada linha, tendo os diversos resultados em linhas independentes. Esta forma de apresentação significa que a altura de cada campo varia consoante está ou não checked.
À direita do botão "Gravar para Todos" deve aparecer um badge (i) que faz aparecer um popup que explica o propósito desta operação  e a forma como é suposto funcionar.

O propósito da operação "Merge" é permitir tornar fácil o consolidar registos que por engano ficaram duplicados. Acontece que os dados corretos para o registo combinado podem estar espalhados pelos diversos registos e a operação de copiar campo a campo é trabalhosa e difícil.
Só faz sentido fazer Merge se estiverem checked pelo menos 2 registos.
A operação Merge tem múltiplos paineis (pelo menos 2) e também faz alinhamento da altura dos campos com base no conteúdo de cada um deles. No caso específico desta operação não é possível escolher apenas 1 painel na select respetiva.
O primeiro painel é um painel do tipo editar, e têm uma estrutura de navegação como acontece no "Multiedit". O Registo que escolhemos neste primeiro painel editar é o registo que vai ser o resultado do merge. Tipicamente escolhemos o registo que tem a maior parte de campos com os valores corretos.
Nos demais paineis (2,3 ou 4) consoante o número de paineis que escolhemos existirem, aparecem paineis em modo similar ao modo "Multiview" com uma nuançe. À esquerda de cada campo aparece uma radiobutton, que por omissão aparece no mesmo registo que foi escolhido no primeiro painel (Editar). É possível assinalar em cada campo de "View" qual dos registos queremos adoptar como valor que é colocado no primeiro painel (Edit), mas apenas se torna definitivo se e quando carregarmos no botão "Juntar Todos num Só". Note-se que esta radiobutton não segue o funcionamento normal das radiobuttons, que usam o atributo "name" para determinar o funcionamento em grupo das várias radiobuttons do mesmo grupo. Nesta operação este comportamento tem de ser simulado, não só porque os radiobuttons podem estar em paineis diferentes, como podem estar numa "página" que nem está visível. Assim, o uiState para esta operação deve guardar (temporariamente) para esta operação para cada campo, qual o id de entre os registos selected que está atualmente escolhido, e ao navegar entre as páginas dos vários paineis, mostrar essa informação nas radiobuttons de forma consistente. Sempre que o utilizador escolhe, pela radiobutton, para um determinado campo, o valor desse campo, esse valor aparece no painel 1 (Editar).
Um caso específico acontece para os campos do tipo relation. Nesse caso, o utilizador deve poder escolher campo a campo (com uma checkbox) se os registos de uma determinada relation devem ser copiados (ou não) para a relation no painel 1 (Edit), mas apenas e quando se carregar no botão "Juntar Todos num Só".
No fundo, ao carregar no botãr "Juntar Todos num Só", para a maioria dos campos o valor a adotar já estará nos campos edit. Apenas os campos relation (e outros kinds que venham a ser acrescentados no futuro) têm de ser copiados para a relation do registo escolhido para destino do merge. Note-se que estes registos devem ter id's únicos (guid), mas pode dar-se o caso de terem o mesmo id, tipicamente para id com números negativos, para registos que ainda não foram sincronizados. Nesse caso informa o utilizador de que não pode Juntar os dados nessa situação, deixando o utilizador escolher se escolhe não transferir os dados daquela relation para a de destino, ou cancelar o Merge e fazer manualmente.
O resultado deve ser, apagar todos os registos, excepto o que resulta do "Merge" e fechar a janela. 
À direita do botão "Juntar Todos num Só" deve aparecer um badge (i) que faz aparecer um popup que explica o propósito desta operação e a forma como é suposto funcionar.
